在做 Agent 系统时，最容易被低估的一件事，不是模型能力，也不是工具调用，而是 **上下文管理**。

很多 Agent 项目在 Demo 阶段看起来都能跑，但一旦进入真实使用场景，很快就会暴露出几个问题：

- 对话一长就“失忆”
- 工具越接越多，决策越混乱
- 明明有历史信息，却不会正确使用
- 上下文塞得太满，成本和延迟越来越高
- 不该记住的东西记住了，该保留的关键信息反而丢了

这些问题本质上都指向同一个核心：**上下文不是越多越好，而是要被有结构地管理。**

这篇文章尝试整理一套适用于 Agent 系统的 **通用上下文管理方案**。它不是某个框架的专属实现，而是一套更偏工程化的抽象，适合用来指导个人项目、产品原型，或者多工具 Agent 的基础架构设计。

## 一、什么是 Agent 的上下文管理

如果把 Agent 看成一个“会思考、会调用工具、会基于历史继续行动的执行体”，那么上下文管理就是：

> 在每次推理或行动前，决定什么信息应该进入模型上下文，以什么形式进入，保留多久，以及在什么阶段被替换或压缩。

这里的重点不是“存信息”，而是“**选择信息**”。

因为模型真正能看到的上下文窗口始终是有限资源。即使窗口足够大，也仍然存在三个现实约束：

1. **成本约束**：上下文越长，推理成本越高
2. **注意力约束**：信息越多，模型越容易抓不住重点
3. **时效约束**：并不是所有历史信息都应该持续生效

所以，一个成熟的 Agent 系统不应该把上下文理解成“聊天记录拼接器”，而应该把它理解成一种 **动态构造的工作记忆**。

## 二、通用上下文管理的核心目标

一个好的上下文管理方案，至少要同时满足四个目标。

### 1. 相关

进入当前上下文的信息，必须和当前任务高度相关。

### 2. 稳定

系统级规则、角色设定、长期约束，不应该随着对话拉长而被冲掉。

### 3. 可压缩

历史不能无限堆积，必须可以被提炼、总结和替换。

### 4. 可追踪

系统要知道一段上下文为什么被放进来，它来自哪里，它什么时候该被淘汰。

如果缺少这些能力，Agent 很容易出现“能跑，但不稳”的状态。

## 三、一个通用可落地的上下文分层模型

我更推荐把 Agent 的上下文拆成五层，而不是只分“系统提示词”和“聊天记录”。

### 1. 系统层上下文

这部分是最稳定的，通常包括：

- 系统角色
- 核心行为准则
- 安全约束
- 输出格式要求
- 工具使用总规则

这一层的特点是：

- 生命周期最长
- 优先级最高
- 不应被普通对话覆盖

你可以把它理解成 Agent 的“操作系统规则”。

#### 示例

- 你是一个偏工程化的 AI 助手
- 回答优先结构化、清晰、可执行
- 涉及外部工具时，必须先判断是否真的需要调用
- 不要在未确认时伪造事实

这一层通常适合固定模板化管理，不应频繁动态拼接。

### 2. 会话层上下文

这部分是当前会话的基础状态，通常包括：

- 当前任务目标
- 当前对话主题
- 当前用户的短期意图
- 本轮内已经达成的中间结论

这一层的特点是：

- 生命周期覆盖当前会话
- 比系统层灵活
- 比记忆层更即时

#### 示例

- 当前会话的任务是设计个人博客
- 技术栈已确定为 Nuxt + Content + Tailwind
- 当前阶段在做信息架构与部署规划
- 用户更希望一步一步引导，而不是一次性给完整成品

这一层非常重要，因为它解决的是“**当前到底在干什么**”。

### 3. 用户画像 / 长期记忆层

这部分是跨会话持久存在的信息，但必须非常克制。

适合进入这一层的内容通常有：

- 长期稳定的偏好
- 持续有效的项目背景
- 对回答方式有明显影响的习惯
- 用户明确要求记住的事实

不适合进入这一层的内容包括：

- 临时情绪
- 短期任务状态
- 一次性问答
- 过于隐私、敏感或易变的信息

#### 示例

- 用户偏好中文交流
- 用户是前端开发工程师
- 用户当前长期项目是个人主页 + Blog 站点
- 用户倾向于渐进式学习和分步骤引导

这一层解决的是“**这个用户长期是谁**”。

### 4. 工作记忆层

这是最像“思考缓存”的一层。它通常保存：

- 当前任务的关键中间结论
- 已尝试过的方法
- 待办项
- 决策理由
- 还没最终落盘但当前阶段必须保留的信息

它的特点是：

- 生命周期中等
- 高度任务相关
- 会不断被更新、压缩、替换

#### 示例

- 当前已经完成博客 MVP 范围界定
- 已选择 Nuxt 而不是 VitePress
- 部署方案确定为 GitHub + Vercel
- 下一步要做 Codex prompts 和项目落地计划

工作记忆层是 Agent “连续做事”的关键。

### 5. 外部检索 / 工具返回层

这部分不是预先长期保存的，而是在当前任务中动态注入的：

- 检索结果
- 文件搜索命中内容
- API 返回值
- 数据库查询结果
- 网页内容摘要

这一层的特点是：

- 动态性最强
- 时效性最强
- 不适合无脑全部保留

正确做法不是原样塞满，而是：

- 只抽取本轮真正相关的结果
- 提取关键字段
- 必要时做结构化摘要
- 明确标注来源

这一层解决的是“**当前外部世界告诉了 Agent 什么**”。

## 四、推荐的上下文拼装顺序

当 Agent 要发起一次推理时，我更推荐使用下面这个顺序拼装上下文：

1. 系统层
2. 当前会话目标层
3. 用户长期记忆层
4. 当前工作记忆层
5. 当前轮真正需要的历史消息
6. 当前工具返回 / 检索结果
7. 当前用户输入

这个顺序背后的逻辑是：

- 先定规则
- 再定目标
- 再补稳定背景
- 再补当前任务状态
- 最后再引入即时输入

这样做能降低一个常见问题：  
**被最近一轮对话带偏，而忘记整个任务主线。**

## 五、历史消息不该怎么存

很多系统在历史管理上会犯两个典型错误。

### 错误一：全量保留

把所有用户消息和模型消息原样堆进上下文。

问题是：

- 成本很快上涨
- 重复信息越来越多
- 模型越来越难抓重点
- 旧的噪音会干扰新的推理

### 错误二：粗暴截断

只保留最后 N 轮对话，前面的全部丢掉。

问题是：

- 长任务容易失去关键背景
- 中间结论断裂
- 需要用户反复重复约束

所以更好的方式是：

> **短历史保留原文，长历史保留摘要。**

也就是把最近几轮消息原样保留，把更早的内容逐步压缩成结构化摘要。

## 六、一个实用的历史压缩策略

一个通用的做法是把历史消息拆成三段：

### 1. 近端消息

例如最近 4 到 8 轮。

特点：

- 保留原文
- 保留语气
- 保留细节
- 用于维持自然连续性

### 2. 中段摘要

把更早的一段历史总结成：

- 已知目标
- 已完成决策
- 已排除方案
- 当前待办

这部分不要写成长段自然语言，最好是结构化。

### 3. 长期沉淀

把真正跨会话有效的信息提取进长期记忆层。

这样，原始长历史可以被安全淘汰，但关键信息不会丢。

#### 推荐摘要格式

```md
## 当前任务摘要
- 当前目标：
- 已确认约束：
- 已完成步骤：
- 已排除方案：
- 下一步建议：
```

这种格式的好处是，模型后续再读时能迅速抓住状态，而不是重新消化一整段自然语言。

## 七、什么时候该触发压缩

压缩不应该等到“上下文已经爆了”才做。更合理的是提前设规则，让系统在几个节点自动触发。

比较常见的触发条件包括：

- 历史 token 接近阈值
- 当前会话完成了一个明显阶段
- 已经做出重要方案决策
- 工具返回内容过长
- 用户主题明显切换

实践里，最稳定的方式通常不是单一阈值，而是“**长度阈值 + 阶段事件**”组合触发。

也就是说：

- 太长了，要压缩
- 即使不长，但阶段已经结束，也应该沉淀成摘要

这样做能避免工作记忆变成一个越滚越长、却没人清理的临时仓库。

## 八、工具调用返回内容怎么处理

很多 Agent 系统的上下文污染，其实不是来自聊天记录，而是来自工具返回。

比如：

- 搜索结果一次返回十几条
- 文件搜索命中几十段
- 网页抓取内容特别长
- API 返回完整 JSON

如果这些内容被原样塞进模型，很容易出现两个后果：

1. 模型读不完
2. 模型虽然读了，但抓不住重点

更合理的做法是把工具返回分成三步处理：

### 1. 过滤

先判断哪些结果和当前任务真正相关。

### 2. 提炼

把结果转换成模型更容易利用的结构，比如：

- 标题
- 关键字段
- 摘要
- 来源
- 可信度

### 3. 限量注入

只把本轮推理真正需要的那部分结果送进上下文，其余内容保留在外部系统里，按需再取。

一个很实用的原则是：

> 不要把工具当作“帮模型看更多东西”的扩容器，而要把工具当作“帮模型带回关键信息”的过滤器。

## 九、记忆系统最容易犯的错

很多人一做 Agent，就很快会想接“长期记忆”。这方向没错，但最容易踩坑。

常见问题包括：

- 什么都记
- 记了但不用
- 用了但没有判断是否还有效
- 记忆和当前任务不相关，却持续干扰回答

所以长期记忆系统至少要解决三个问题：

### 1. 写入门槛

不是每次对话都值得进入长期记忆。

### 2. 检索门槛

不是每条记忆都该在每轮被召回。

### 3. 失效机制

有些记忆应该过期，有些记忆应该被更新。

如果没有这三层约束，记忆系统只会从“增强能力”变成“长期噪音源”。

## 十、一个更实用的工程化架构

如果把它真正落到系统里，一个比较稳的工程化结构通常会长这样：

### 输入侧

- 当前用户输入
- 当前会话状态
- 长期记忆召回结果
- 工具调用候选信息

### 编排侧

- 上下文选择器
- 历史压缩器
- 记忆写入判断器
- 工具结果提炼器

### 输出侧

- 当前推理上下文
- 可沉淀摘要
- 记忆候选项
- 下一步行动决策

这里一个很关键的认知是：

> 上下文管理不是 prompt 工程的附属品，而是 Agent 编排层的核心能力。

如果系统没有这层能力，模型再强，表现也会随着任务拉长快速波动。

## 十一、一个简单的落地原则

如果你现在就在做个人项目，没必要一开始就把系统做得特别复杂。更推荐先抓住下面四条：

1. 系统规则固定化
2. 当前会话目标单独维护
3. 工作记忆单独维护
4. 长历史定期压缩

只要先把这四条做好，Agent 的稳定性通常就会明显提升。

真正的复杂度，往往不是来自模型本身，而是来自：

- 哪些信息要保留
- 哪些信息要丢掉
- 哪些信息要被重新组织

这件事做得越早，后面的工具接入、记忆扩展和多阶段任务执行就越稳。

## 结语

Agent 的上下文管理，本质上是在解决一个很朴素但非常关键的问题：

> 在有限窗口里，怎样让模型始终看到“现在最该看到的东西”。

它不是一个只靠“加大上下文窗口”就能自动解决的问题，也不是简单的历史拼接问题，而是一套需要持续设计、压缩、选择和更新的工程能力。

如果一个 Agent 想要从“能用”走向“稳定可用”，上下文管理往往不是配角，而是最值得优先打磨的底层部分。
